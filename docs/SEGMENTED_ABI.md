# Segmented ABI Design

> **Status:** Implemented in v0.15.0 (8-byte descriptors), upgraded to
> capacity-aware 12-byte descriptors in v0.17.0.
> See `jiminy-core::account::segment` and the `segmented_layout!` macro.

## Problem

`zero_copy_layout!` produces fixed-size `#[repr(C)]` structs. Every
instance of a given account type occupies exactly `LEN` bytes. This
works for most account types (vaults, pools, config), but some programs
need variable-length data:

- Order books with a dynamic number of orders
- Position arrays of unbounded size
- Token metadata with optional extended fields
- Accounts that grow via `realloc` over their lifetime

Today programs handle this with manual offset arithmetic or by
allocating a separate account for each dynamic element. Both approaches
are error-prone and waste rent.

## Existing Building Block

Jiminy already provides `ZeroCopySlice` - a length-prefixed,
alignment-1 slice overlay:

```
[len: u32 LE][element_0][element_1]...[element_N-1]
```

This handles a single dynamic array appended to a fixed prefix.
Segmented ABI generalizes this to **multiple** variable-length sections
within one account.

## Design: Segment Descriptors

A segmented account has three regions:

```
┌──────────────┬──────────────────┬──────────────────────────┐
│ Fixed Header │  Segment Table   │     Segment Data         │
│  (≥16 bytes) │  (N × 12 bytes)  │  (variable per segment)  │
└──────────────┴──────────────────┴──────────────────────────┘
```

### Fixed Header

The standard 16-byte Jiminy `AccountHeader`. The `flags` field
encodes the number of segments (or a sentinel indicating "segmented
layout").

### Segment Table

Immediately after the fixed prefix. Each entry is 12 bytes:

```
Byte   Field          Type      Description
──────────────────────────────────────────────────────────
0-3    offset         u32 LE    Byte offset from account start
4-5    count          u16 LE    Number of live elements
6-7    capacity       u16 LE    Maximum element capacity
8-9    element_size   u16 LE    Size of each element in bytes
10-11  flags          u16 LE    Reserved (must be zero)
──────────────────────────────────────────────────────────
```

The `capacity` field defines the reserved region for this segment
(`capacity × element_size` bytes). The first `count` elements are live;
slots `count..capacity` are zeroed spare capacity. Invariant:
`count ≤ capacity`.

The segment table is fixed once the account is created. You declare
the number of segments at account creation time. Individual segments
can grow (via `realloc`) but the number of segments doesn't change.

### Segment Data

Each segment's data starts at the offset specified in its table entry.
Elements are laid out contiguously with no padding between them.
Segments are ordered by offset (no overlap, no gaps between segments
unless explicitly padded).

## Wire Format Example

An order book account with a fixed config region, a dynamic bids
array, and a dynamic asks array:

```rust,ignore
// Fixed portion (declared via zero_copy_layout!)
zero_copy_layout! {
    pub struct OrderBookFixed, discriminator = 5, version = 1 {
        header:     AccountHeader = 16,
        market:     Address       = 32,
        base_mint:  Address       = 32,
        quote_mint: Address       = 32,
    }
}

// Segmented layout: fixed prefix + dynamic segments.
segmented_layout! {
    pub struct OrderBook, discriminator = 5, version = 1 {
        header:     AccountHeader = 16,
        market:     Address       = 32,
        base_mint:  Address       = 32,
        quote_mint: Address       = 32,
    } segments {
        bids: Order = 48,       // dynamic-length segment
        asks: Order = 48,       // dynamic-length segment
    }
}
```

Wire layout (example with 3 bids, 2 asks, Order = 48 bytes):

```
Offset  Size  Content
─────────────────────────────────────
0       16    AccountHeader
16      32    market (Address)
48      32    base_mint (Address)
80      32    quote_mint (Address)
112     12    Segment descriptor: bids  { offset: 136, count: 3, capacity: 3, elem_size: 48, flags: 0 }
124     12    Segment descriptor: asks  { offset: 280, count: 2, capacity: 2, elem_size: 48, flags: 0 }
136    144    bids[0..3] (3 × 48)
280     96    asks[0..2] (2 × 48)
─────────────────────────────────────
Total: 376 bytes
```

## Layout ID for Segmented Accounts

The hash input extends the existing convention:

```
jiminy:v1:<Name>:<version>:<fixed_fields>,seg:<seg_name>:<element_type>:<element_size>,...
```

Segments use the `seg:` prefix to distinguish them from fixed fields.
This ensures that adding, removing, or reordering segments changes the
`LAYOUT_ID`.

## Access API

```rust,ignore
// ── Constants generated by the macro ──────────────────────────────
OrderBook::SEGMENT_COUNT;      // 2
OrderBook::FIXED_LEN;          // 112 (header + market + base_mint + quote_mint)
OrderBook::TABLE_OFFSET;       // 112 (same as FIXED_LEN)
OrderBook::DATA_START_OFFSET;  // 136 (FIXED_LEN + 2 × 12-byte descriptors)
OrderBook::MIN_ACCOUNT_SIZE;   // 136 (fixed + table, zero elements)
OrderBook::SEGMENTED_LAYOUT_ID;// [u8; 8] including segment hashes

// Named segment indices (generated from segment names).
OrderBook::bids;  // 0
OrderBook::asks;  // 1

// ── Initialize ───────────────────────────────────────────────────
// Compute the total account size for given element counts.
let size = OrderBook::compute_account_size(&[3, 2])?; // 3 bids, 2 asks

// After creating the account, initialize the segment table.
OrderBook::init_segments(&mut data, &[3, 2])?;

// ── Read access ──────────────────────────────────────────────────
// Get a typed immutable view over a segment.
let bids = OrderBook::segment::<Order>(&data, OrderBook::bids)?;
let first_bid: Order = bids.read(0)?;

// Iterate over segment elements.
for bid in bids.iter() {
    // bid: Order (read by copy)
}

// ── Write access ─────────────────────────────────────────────────
// Initialize with capacity for dynamic push/remove workflows.
// Offsets are spaced by capacity; counts start at 0.
let size = OrderBook::compute_account_size(&[100, 100])?;
// ... create account with `size` bytes ...
OrderBook::init_segments_with_capacity(&mut data, &[100, 100])?;

// Push an element (appends, increments descriptor count).
// Push checks the next segment's offset to prevent overlap.
OrderBook::push::<Order>(&mut data, OrderBook::bids, &new_order)?;

// Swap-remove an element (O(1), does not preserve order).
let removed: Order = OrderBook::swap_remove::<Order>(
    &mut data, OrderBook::bids, 0,
)?;

// Get a mutable typed view for element-level writes.
let mut asks = OrderBook::segment_mut::<Order>(&mut data, OrderBook::asks)?;
asks.set(0, &updated_ask)?;

// ── Pre-populated init ───────────────────────────────────────────
// Or init with known counts when all elements are written upfront.
OrderBook::init_segments(&mut data, &[3, 2])?;

// ── Validation ───────────────────────────────────────────────────
// Full segment table validation (bounds, sizes, no overlaps).
OrderBook::validate_segments(&data)?;
```

### `init_segments` vs `init_segments_with_capacity`

| | `init_segments(counts)` | `init_segments_with_capacity(capacities)` |
|---|---|---|
| **Use when** | All elements are written upfront | Elements are pushed/removed dynamically |
| **Counts** | Set to `counts[i]` | Set to `0` |
| **Capacity** | Set to `counts[i]` (tight fit) | Set to `capacities[i]` |
| **Offsets** | Spaced by `counts[i] * elem_size` | Spaced by `capacities[i] * elem_size` |
| **Push safe?** | No — segment is already full (`count == capacity`) | Yes, up to `capacities[i]` elements |

## Bounds Checking & Overflow

Every segment access validates:

1. Segment index < segment count (from header flags).
2. `count ≤ capacity` for the descriptor.
3. `offset + capacity * element_size` (reserved region) does not exceed
   account data length.
4. No segment's reserved region overlaps with another segment's reserved
   region.
5. Element size from the segment table matches the expected
   `FixedLayout::SIZE` of the element type.
6. `push` checks `count < capacity` before writing. When `count ==
   capacity`, `push` returns `ProgramError::AccountDataTooSmall`.
   The caller must `realloc` the account and update the descriptor's
   capacity before pushing.

All offset arithmetic uses checked operations to prevent overflow on
32-bit targets.

## Migration Path

Programs can migrate from fixed to segmented layouts:

1. Deploy v1 with `zero_copy_layout!` (fixed size).
2. When dynamic data is needed, deploy v2 with `segmented_layout!`.
   Bump the version. The new layout's `LAYOUT_ID` differs from v1
   because of the `seg:` hash entries.
3. Existing v1 accounts can be migrated in an instruction that
   `realloc`s, writes the segment table, and updates the header
   version + layout_id.

The `extends` mechanism still works: a segmented v2 can extend a fixed
v1 as long as the fixed fields are a byte-compatible prefix.

## Safety Properties

- **No raw pointer arithmetic.** Segment offsets are `u32` values
  resolved to `&[u8]` slices via safe Rust indexing.
- **`repr(C)` elements only.** Segment elements must implement `Pod`
  + `FixedLayout`, same as `ZeroCopySlice` today.
- **Alignment-1 access.** All reads go through `LeU64`, `FieldRef`,
  etc. No alignment requirements beyond byte alignment.
- **Deterministic validation.** Call `validate_segments()` after loading
  to check the segment table. Corrupted descriptors produce `ProgramError`,
  never UB. This is an explicit call, not implicit in `load()`.

## Design Decisions (Frozen - v0.17.0)

The following decisions are locked. They define the segmented ABI
contract and will not change without a major version bump.

1. **12-byte descriptors with explicit capacity.** Each descriptor is
   12 bytes: `[offset:u32][count:u16][capacity:u16][element_size:u16]
   [flags:u16]`. The `capacity` field defines the reserved region;
   `count` tracks live elements. Invariant: `count ≤ capacity`.
2. **Max segments: 8.** The segment count is capped at
   `MAX_SEGMENTS = 8` (defined in `segment.rs`). This bounds the
   segment table to 96 bytes, keeps validation O(1), and prevents
   excessive rent overhead. Programs needing more than 8 dynamic
   regions should split across multiple accounts.
3. **No nested segments.** The model is intentionally flat. Nest by
   having segment elements reference other accounts instead.
4. **No sparse segments.** Use `swap_remove` for O(1) deletion.
   `swap_remove` zeroes the vacated slot (last element's former
   position) and decrements the descriptor's `count` field.
   Order is not preserved. This is a documented trade-off for O(1)
   removal without fragmentation.
5. **No auto-realloc.** `push` does **not** auto-realloc. When
   `count == capacity`, `push` returns an error. The caller must
   `realloc` the account and update the descriptor's `capacity`
   before pushing. This keeps the API predictable and avoids
   hidden CPI costs.
6. **Segment count is immutable after init.** The number of segments
   is fixed at account creation via `init_segments`. Adding or removing
   segment slots requires migration to a new layout version.
7. **Capacity is encoded in the account, not inferred.** Unlike other
   Solana crates that derive capacity from buffer length, Jiminy
   stores capacity explicitly in each descriptor. This enables
   multi-segment accounts where buffer length alone cannot determine
   per-segment capacity.

## Relationship to Other Innovations

| Innovation | Interaction |
|---|---|
| Le* types (Phase 1) | Segment elements use Le* types for alignment-1 safety |
| Borrow-splitting (Phase 2) | `split_fields` works for the fixed portion; segment access is separate |
| Manifest (Phase 3) | `LayoutManifest` gains a `segments` field describing dynamic regions |
| Interface (Phase 4) | `jiminy_interface!` can declare foreign segmented layouts for read-only access |
