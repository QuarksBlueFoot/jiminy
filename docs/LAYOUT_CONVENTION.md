# Jiminy Account Layout Convention (v2)

A strict, copy-pasteable convention for Solana account data layouts in
Pinocchio programs using Jiminy.

## The Header

Every Jiminy-managed account starts with a **16-byte fixed header**:

```
Byte   Field          Type      Description
──────────────────────────────────────────────────────────
0      discriminator  u8        Account type tag (unique per program)
1      version        u8        Schema version. Bump when layout changes
2-3    flags          u16 (LE)  Application-defined bitfield
4-11   layout_id      [u8; 8]   ABI fingerprint (first 8 bytes of SHA-256)
12-15  reserved       [u8; 4]   Must be zero (future use)
──────────────────────────────────────────────────────────
```

Payload fields follow immediately after byte 15.

### Layout ID

The `layout_id` field is a deterministic 8-byte ABI fingerprint derived
from the struct name, version, field names, canonical types, and byte
sizes. It is computed at compile time as:

```
sha256("jiminy:v1:<Name>:<version>:<field>:<canonical_type>:<size>,...")[..8]
```

Canonical type mappings:

| Rust type       | Canonical name |
|-----------------|----------------|
| `u8`–`u128`    | `"u8"`–`"u128"` |
| `i8`–`i128`    | `"i8"`–`"i128"` |
| `bool`          | `"bool"`       |
| `Address`       | `"pubkey"`     |
| `AccountHeader` | `"header"`     |
| Other ident     | `stringify!()` |

When any field is renamed, retyped, resized, or reordered, the hash
changes, giving on-chain programs a cheap way to reject accounts whose
layout doesn't match the expected schema.

## Rules

1. **Discriminator is immutable.** Once an account type is deployed, its
   discriminator value never changes.

2. **Version must increment** whenever the payload layout changes. This
   includes adding fields, removing fields, or changing field sizes.

3. **Fields never move.** Once a field is assigned an offset within a
   version, that offset is permanent for that version. Append new fields
   at the end.

4. **Account size never decreases.** If you need fewer fields, leave the
   old bytes as reserved padding. Programs that encounter an older, larger
   account should still work.

5. **Zero-init before writing.** Always call `zero_init()` on new account
   data before writing the header. This ensures flags, reserved bytes,
   and any trailing padding start as zero.

## Declaring a Layout

Use the `zero_copy_layout!` macro to declare an account struct. The macro
generates the `#[repr(C)]` struct, `Pod` + `FixedLayout` impls, overlay
methods, and compile-time constants (`DISC`, `VERSION`, `LEN`, `LAYOUT_ID`):

```rust
use jiminy::prelude::*;

zero_copy_layout! {
    /// On-chain vault account.
    pub struct Vault, discriminator = 1, version = 1 {
        header:    AccountHeader = 16,
        balance:   u64           = 8,
        authority: Address       = 32,
    }
}

// Constants generated by the macro:
const _: u8 = Vault::DISC;              // 1
const _: u8 = Vault::VERSION;           // 1
const _: usize = Vault::LEN;            // 56
const _: [u8; 8] = Vault::LAYOUT_ID;    // SHA-256 fingerprint
```

## Using the Header in Code

### Writing (init) - recommended

```rust
use jiminy::prelude::*;

// init_account! creates the account, zeroes data, and writes the header.
init_account!(payer, vault, program_id, Vault)?;

// Set fields via overlay.
let mut raw = vault.try_borrow_mut()?;
let v = Vault::overlay_mut(&mut raw)?;
v.balance = 0;
v.authority = authority;
```

### Writing (init) - manual

```rust
use jiminy::prelude::*;

let mut raw = account.try_borrow_mut()?;
zero_init(&mut raw);
write_header(&mut raw, Vault::DISC, Vault::VERSION, &Vault::LAYOUT_ID)?;
let v = Vault::overlay_mut(&mut raw)?;
v.balance = initial_balance;
v.authority = authority;
```

### Reading (validate + overlay)

```rust
use jiminy::prelude::*;

let data = account.try_borrow()?;
let vault = Vault::load_checked(&data)?;
let auth: &Address = &vault.authority;
```

### Tiered loading

`zero_copy_layout!` generates per-struct tiered loading methods that
combine validation and borrowing in a single call:

```rust
// Tier 1: Verified (recommended).
// Validates: owner + disc + version + layout_id + exact size.
// Returns VerifiedAccount whose get() is infallible.
let verified = Vault::load(account, program_id)?;
let vault = verified.get();

// Tier 1 (mutable): same checks, returns VerifiedAccountMut.
let mut verified = Vault::load_mut(account, program_id)?;
let vault = verified.get_mut();

// Tier 2: Foreign Verified (cross-program read).
// Validates: owner + layout_id + exact size (skips disc/version).
let verified = Vault::load_foreign(account, &other_program_id)?;
let vault = verified.get();

// Tier 3: Compatibility (version migration).
// Validates: owner + disc + version + size (skips layout_id).
let data = validate_version_compatible(account, program_id, Vault::DISC, 1, Vault::LEN)?;

// Tier 4: Unsafe unchecked (requires manual validation).
let data = account.try_borrow()?;
let vault = unsafe { Vault::load_unchecked(&data)? };

// Tier 5: Unverified overlay (indexers/tooling).
// No ABI guarantees. Returns (overlay, validated) where validated is
// true if header matched.
let data = account.try_borrow()?;
let (vault, validated) = Vault::load_unverified_overlay(&data)?;
```

See [SAFETY_MODEL.md](SAFETY_MODEL.md) for the full trust tier model.

### Identity vs. Authorization

`layout_id` proves **ABI identity**: that the on-chain bytes match your
expected struct layout. It does **not** prove that the account is
authorized, trustworthy, or safe to act on.

- **Owner check is never optional** when ownership matters. A matching
  `layout_id` does not replace `account.owner() == program_id`.
- **`load_foreign` proves layout compatibility** with a foreign program's
  account. It does not prove that the foreign program is honest, audited,
  or bug-free. You are choosing to trust that program's ABI surface.
- **`validate_version_compatible` is the weakest trust tier** that still
  checks ownership. It accepts any version ≥ yours, which means the
  remote program may have added fields you have never audited.

Design your access control around ownership and signatures, not around
ABI fingerprints. See [SAFETY_MODEL.md](SAFETY_MODEL.md) §7 for the
full trust analysis.

### Version migration

```rust
let data = account.try_borrow()?;
let version = read_version(&data)?;
match version {
    1 => { /* read v1 fields */ }
    2 => { /* read v1 fields + new v2 fields */ }
    _ => return Err(ProgramError::InvalidAccountData),
}
```

## Layout Lint (Test-Time)

Add this pattern to your program's test suite to catch accidental layout
drift:

```rust
#[cfg(test)]
mod layout_tests {
    use super::*;

    // If you change VAULT_LEN, you MUST bump VAULT_VERSION.
    const EXPECTED_LEN: usize = 56;
    const EXPECTED_VERSION: u8 = 1;

    #[test]
    fn vault_layout_stable() {
        assert_eq!(Vault::LEN, EXPECTED_LEN, "layout changed - bump VERSION");
        assert_eq!(Vault::VERSION, EXPECTED_VERSION);
    }

    #[test]
    fn vault_offsets_valid() {
        // Header is 16 bytes.
        assert_eq!(HEADER_LEN, 16);
        // Total: header(16) + balance(8) + authority(32) = 56.
        assert_eq!(HEADER_LEN + 8 + 32, Vault::LEN);
    }

    #[test]
    fn layout_id_stable() {
        // Pin the layout ID to catch silent ABI changes.
        let expected: [u8; 8] = Vault::LAYOUT_ID;
        assert_eq!(Vault::LAYOUT_ID, expected);
    }
}
```

This test will fail CI if someone changes the account size without
updating the version constant, catching layout drift before it reaches
mainnet.

## Layout Inheritance (`extends`)

When evolving an account schema, use `extends = ParentType` to enforce
that the new version is a byte-compatible superset of the old:

```rust
zero_copy_layout! {
    pub struct VaultV2, discriminator = 1, version = 2, extends = Vault {
        header:    AccountHeader = 16,
        balance:   u64           = 8,
        authority: Address       = 32,
        fee_bps:   u16           = 2,
    }
}
```

The macro asserts at compile time:

1. `VaultV2::DISC == Vault::DISC`: same account type
2. `VaultV2::LEN >= Vault::LEN`: append-only (never shrink)
3. `VaultV2::VERSION > Vault::VERSION`: strictly newer

If any assertion fails, the build fails. This catches accidental
discriminator mismatches and shrinking layouts before they reach mainnet.

See [ABI_VERSIONING.md](ABI_VERSIONING.md) for the full versioning
strategy.

## Const Field Offsets

The `zero_copy_layout!` macro generates compile-time offset constants
for every field. These are byte offsets from the start of the account
data:

```rust
zero_copy_layout! {
    pub struct Vault, discriminator = 1, version = 1 {
        header:    AccountHeader = 16,
        balance:   u64           = 8,
        authority: Address       = 32,
    }
}

// Generated constants:
const _: usize = Vault::OFFSET_HEADER;    // 0
const _: usize = Vault::OFFSET_BALANCE;   // 16
const _: usize = Vault::OFFSET_AUTHORITY; // 24
```

Use these for low-level byte manipulation when `overlay()` is
unnecessary, or in tests to pin offsets against accidental drift.

## Borrow-Splitting (`split_fields`)

The macro also generates `split_fields` and `split_fields_mut` methods
that decompose borrowed data into per-field `FieldRef` / `FieldMut`
wrappers. Each wrapper holds a non-overlapping subslice - the borrow
checker sees independent references:

```rust
// Immutable: read multiple fields without aliasing.
let (header, balance, authority) = Vault::split_fields(&data)?;
let bal: u64 = balance.read_u64();
let auth: &Address = authority.as_address();

// Mutable: write to multiple fields simultaneously.
let (_hdr, balance, authority) = Vault::split_fields_mut(&mut data)?;
balance.write_u64(1000);
authority.copy_from(new_authority.as_ref());
```

`FieldRef` provides typed readers: `read_u64()`, `read_u32()`,
`read_u16()`, `read_u8()`, `read_bool()`, `read_i64()`, `read_i32()`,
`read_i16()`, `read_i128()`, `read_u128()`, `read_address()`,
`as_address()`, `as_bytes()`, `len()`.

`FieldMut` provides the matching writers: `write_u64()`, `write_u32()`,
`write_u16()`, `write_u8()`, `write_bool()`, `write_i64()`,
`write_address()`, `copy_from()`, `as_bytes()`, `as_bytes_mut()`,
`len()`.

## Padding Discipline

Any reserved space in an account layout must be declared explicitly as
a named field. **Do not rely on implicit compiler padding as semantic
storage.**

```rust
// ✓ Good: reserved bytes are explicit
zero_copy_layout! {
    pub struct Order, discriminator = 5, version = 1 {
        header:    AccountHeader = 16,
        price:     u64           = 8,
        _reserved: u64           = 8,   // explicit padding
        owner:     Address       = 32,
    }
}

// ✗ Bad: relying on compiler-inserted padding between fields
//   (this would compile but the padding bytes are not part of the ABI)
```

`#[repr(C)]` guarantees field order but allows alignment-driven padding.
Explicit reserved fields are part of the ABI contract and survive across
compiler versions and targets.

## FAQ

**Q: Why not use Anchor's 8-byte discriminator?**
A: Anchor uses the first 8 bytes of `sha256("account:TypeName")` as a
discriminator. This is safe but costs 7 extra bytes per account. Jiminy
uses a single `u8` discriminator (256 account types per program) and puts
the ABI fingerprint in a separate `layout_id` field, giving both cheap
type tagging and strong schema validation.

**Q: What changed from v1 to v2 of this convention?**
A: The header grew from 8 to 16 bytes. The old `data_len` field (bytes
4-7, u32) was replaced by `layout_id` (bytes 4-11, [u8; 8]) and
`reserved` (bytes 12-15, [u8; 4]). The `flags` field expanded from u8 to
u16. The functions `write_header_with_len` and `read_data_len` were
removed.

**Q: What about backward compatibility?**
A: Programs should always check `version >= MIN_VERSION` using
`check_header`. If a program encounters a version it doesn't understand
(too new), it should return `InvalidAccountData`. If it encounters an
older version, it can either migrate in-place or read the old layout.

**Q: Do I have to use this header?**
A: No. Jiminy's check functions work with any layout. The header is a
*recommended convention* that gives you versioning, flags, and layout
validation for free. If you're porting an existing program that uses a
different discriminator scheme, keep using it.

**Q: Do all field types need to implement `Copy`?**
A: Yes. The overlay struct derives `Copy` (required by `Pod`). Enable
pinocchio's `"copy"` feature to make `Address` implement `Copy`. Jiminy's
workspace dependency already includes this feature.
